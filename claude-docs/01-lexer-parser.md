# Lexer and Parser

## Files
- `src/yylex.c` — hand-written lexer
- `src/yyparse.y` — YACC grammar
- `src/op.h` — operator precedence/associativity definitions
- `src/Mult-op.awk` — generates `op.sed` from `op.h`
- `src/Assoc.sed` — injects `%left`/`%right` declarations

---

## Shared Conventions (`defs.h`)

The C source makes heavy use of macros that simulate structured control flow:

```c
#define when     break; case
#define or       : case
#define otherwise break; default
#define repeat   for (;;)
#define until(c) if (c) break
#define local    static          /* hidden in non-DEBUG builds */
#define global   /* nothing */   /* explicit marker for external linkage */
```

The types `Bool`, `Natural`, `Byte`, `SBool` are defined here. All internal strings are **interned** via `newstring()`/`newnstring()` — every distinct string value has exactly one heap address, so string equality is pointer equality.

## Lexer (`src/yylex.c`)

### Entry point: `yylex()`

Called by YACC whenever it needs a token. Returns an integer token code and fills `yylval`.

### Token categories

| Category | Example | Condition | Token |
|----------|---------|-----------|-------|
| EOF / newline | end of input | `hope_getline()` returns false | `EOF` |
| Numeric literal | `42`, `3.14e-2` | starts with digit | `NUMBER` |
| Alpha identifier | `max`, `fib'` | letter or `_` | `IDENT` or reserved keyword |
| Punctuation | `(`, `)`, `,`, `;`, `[`, `]` | one of these 6 | the character itself |
| Comment | `! this is a comment` | `!` | skipped entirely |
| Character literal | `'a'`, `'\n'` | `'` | `CHAR` |
| String literal | `"hello"` | `"` | `LITERAL` |
| Operator identifier | `+`, `<=`, `::`, `<>` | sequence of graphic chars | `IDENT` or operator token |

**Numeric literals** consume `{digit}+` and optionally (if compiled with `REALS`) `.{digit}+` and `[eE][+-]?{digit}+`. Leading signs are not part of the literal. Value is parsed via `atoNUM()`.

**Alpha identifiers** are `({letter}|_)({digit}|{letter}|_)*'*`. Note trailing single-quotes are part of the name (e.g. `alpha'`).

**Graphic (operator) identifiers** consume any non-space, non-letter, non-digit, non-punctuation characters until a delimiter. This means things like `->`, `<=`, `<>`, `::`, `++` are all single tokens.

**Comments** begin with `!` and run to the end of the line. The lexer simply scans forward until `\0` (end of current line buffer).

### Reserved word lookup: `lookup(s)`

After scanning an identifier, `lookup()` checks:
1. Is it a reserved word? (linear search through `reserved[]` by pointer equality, since strings are interned)
2. Is it a declared operator? (`op_lookup(s)`)
3. Otherwise return `IDENT`.

### Reserved words table

All reserved words are pre-interned in `init_lex()`. The full table (from `yylex.c`):

```
++       → OR         ---      → VALOF
:        → ':'        <=       → IS
==       → DEFEQ      =>       → GIVES
abstype  → ABSTYPE    data     → DATA
dec      → DEC        display  → DISPLAY
edit     → EDIT       else     → ELSE
end      → END        exit     → EXIT
if       → IF         in       → IN
infix    → INFIX      infixr   → INFIXR
infixrl  → INFIXR     lambda   → LAMBDA
let      → LET        letrec   → LETREC
module   → MODSYM     mu       → MU
nonop    → NONOP      private  → PRIVATE
pubconst → PUBCONST   pubfun   → PUBFUN
pubtype  → PUBTYPE    save     → SAVE
then     → THEN       to       → TO
type     → TYPESYM    typevar  → TYPEVAR
use      → USES       uses     → USES
where    → WHERE      whererec → WHEREREC
write    → WRITE      |        → '|'
\        → LAMBDA
```

Compatibility aliases: `use` = `uses`, `infixrl` = `infixr`, `end`/`module`/`pubconst`/`pubfun`/`pubtype`/`nonop` are accepted but mostly ignored.

UTF-8 support: if compiled with `UTF_LIBS`, the Greek letters λ (lambda), μ (mu), and arrows ⊕, ⇐, ⇒, ≜ are also recognised as their ASCII keyword equivalents.

### Operator token encoding

When `lookup()` finds a declared operator, it computes a YACC token number:

```c
return (op->op_prec - MINPREC) * NUM_ASSOC + (int)(op->op_assoc) + BIN_BASE + 1;
```

This encodes precedence (1–9) and associativity (LEFT=0, RIGHT=1) into a single integer. The token numbers form a range `[BIN_BASE+1, BIN_BASE + (MAXPREC-MINPREC+1)*NUM_ASSOC]`, one token per (precedence, associativity) pair. YACC is told about all of them via the `%left`/`%right` declarations generated by `Assoc.sed`.

### Character literals

C-style escapes: `\a`, `\b`, `\f`, `\n`, `\r`, `\t`, `\v`, `\xNN` (hex), `\XNNNN` (long hex), `\OOO` (octal). Other escaped characters are returned as-is.

### String literals

Accumulated into a static 100-character buffer. C-style escapes are supported. Strings exceeding 100 characters trigger a lexer error. The result is a `Text*` (start pointer + length) returned in `yylval.textval`.

### Interactive error recovery

When the lexer reaches end-of-line in interactive mode while YACC is recovering from an error (`recovering()` returns true), it synthesises a `;` token. This allows the parser to resynchronise at statement boundaries after a syntax error without requiring the user to type a semicolon explicitly.

## Parser (`src/yyparse.y`)

### Grammar structure

The grammar is processed through two sed filters before being fed to YACC:

1. **`op.sed`** (generated by `Mult-op.awk` from `op.h`): Replicates every grammar rule or token declaration containing `BINARY` once for each (precedence × associativity) combination.
2. **`Assoc.sed`**: Changes each replicated `%token BINARY` line into the appropriate `%left` or `%right` declaration.

The result is that each infix operator has its own YACC token with the correct precedence and associativity for shift-reduce disambiguation.

### Precedence table (weakest to strongest)

```
END                     (for optional lambda end keyword)
MU                      (mu-type / mu-expression)
,                       (right assoc: pairs and tuples)
|                       (lambda rule separator)
IN                      (let/letrec body)
WHERE WHEREREC          (where clause)
ELSE                    (if-then-else)
[operators prec 1..9]   (user-defined, left or right assoc)
APPLY IDENT NUMBER ...  (function application, left assoc, tightest)
```

Application is left-associative and binds tighter than all operators. This means `f x + 1` parses as `(f x) + 1`.

### Top-level structure

```
lines → (line)*
line  → cmd ';'  |  error ';'
```

Every statement ends with `;`. Syntax errors are recovered by discarding up to the next `;`.

### Command productions

| Production | Action |
|-----------|--------|
| `TYPEVAR newtv...` | Declares new type variables |
| `INFIX ident,... : n` | Declares left-assoc operators at precedence n |
| `INFIXR ident,... : n` | Declares right-assoc operators at precedence n |
| `ABSTYPE newtype` | Declares an abstract type |
| `DATA newtype == constypelist` | Defines an algebraic data type |
| `TYPESYM newtype == type` | Defines a type synonym |
| `PRIVATE` | Marks subsequent defs as private |
| `DEC decl` | Type declaration |
| `simple_decl` | Abbreviated type declaration (no `dec` keyword) |
| `VALOF expr IS exprbody` | Full equation definition (`--- f pat <= expr`) |
| `expr IS exprbody` | Abbreviated equation (`f pat <= expr`) |
| `expr DEFEQ exprbody` | Variant abbreviated equation (`f pat == expr`) |
| `exprbody` | Interactive evaluation |
| `WRITE expr [TO "file"]` | Lazy streaming output |
| `DISPLAY` | Display current definitions |
| `USES ident,...` | Load modules |
| `SAVE ident` | Save session to module |
| `EDIT [ident]` | Invoke editor (Unix only) |
| `EXIT` | Exit interpreter |

### Expression grammar

```
expr ::=
    ident                           identifier reference
  | NUMBER                          numeric literal
  | LITERAL                         string literal (list of chars)
  | CHAR                            character literal
  | '(' expr binop ')'              pre-section: (e op) = \x => e op x
  | '(' binop expr ')'              post-section: (op e) = \x => x op e
  | '(' exprbody ')'                parenthesised expression
  | '[' exprlist ']'                list literal → desugared to :: / nil
  | '[]'                            nil
  | expr expr                       application (left assoc, tightest)
  | expr BINARY expr                infix operator application
  | LAMBDA rulelist [END]           anonymous function (multi-clause)
  | IF expr THEN exprbody ELSE expr conditional
  | LET exprbody == exprbody IN expr  non-recursive let
  | LETREC tuple == exprbody IN expr  recursive let (irrefutable LHS)
  | expr WHERE exprbody == expr     where clause
  | expr WHEREREC tuple == expr     recursive where
  | MU tuple GIVES expr             mu-expression (infinite type fixpoint)

exprbody ::= expr | expr ',' exprbody    (comma = pair/tuple constructor)
exprlist  ::= expr | expr ',' exprlist   (list literal elements)
rulelist  ::= formals '=>' expr ('|' rulelist)?
formals   ::= exprbody                   (pattern)
```

**Key design point**: The grammar uses `exprbody` (which includes commas) for patterns in lambda and let, meaning `lambda (x, y) => ...` parses the pattern as a pair `(x, y)`.

**Operator sections**: `(e op)` and `(op e)` are syntactic sugar. The parser calls `presection(op, e)` and `postsection(op, e)` respectively, which construct `lambda x => e op x` and `lambda x => x op e`.

**List literals**: `[a, b, c]` is desugared during parsing to `a :: (b :: (c :: nil))` via repeated `apply_expr(e_cons, pair_expr(e, rest))`.

**Infix operators as function applications**: `a + b` is parsed as `(+)(a, b)`, i.e. `apply_expr(id_expr("+"), pair_expr(a, b))`.

### Type grammar

```
type ::=
    ident typeargs                  named type, possibly applied
  | ident '(' typepair ')'          type with tuple argument
  | type BINARY type                infix type constructor (e.g. alpha # beta)
  | MU mu_tv '=>' type              regular (recursive) type
  | '(' type ')'

typeargs ::= (typearg)*
typearg  ::= ident | '(' type ')'
```

The `MU` form introduces a recursive type binding: `mu s => alpha # s` is an infinite type unrolled from `s = alpha # s`. This is a Paterson extension not present in Imperial HOPE.

### Pattern grammar

Patterns and expressions share the same grammar (`exprbody`, `expr`). The compiler distinguishes them later. Variables are identifiers that are not declared data constants or constructors.

The `pat + k` abbreviation (matches `succ` applied k times) is handled by the expression grammar: it looks like `expr + NUMBER` to the parser.

### Actions: what the parser does

The parser drives the entire pipeline. Actions in grammar rules call:

| Function | Purpose |
|----------|---------|
| `tv_declare(name)` | Register a new type variable |
| `op_declare(name, prec, assoc)` | Register an infix operator |
| `new_deftype(name, pair, tvlist)` | Create a new type definition node |
| `abstype(deftype)` | Register an abstract type |
| `type_syn(deftype, type)` | Register a type synonym |
| `decl_type(deftype, conslist)` | Register a data type with constructors |
| `decl_value(name, qtype)` | Register a value identifier with its declared type |
| `def_value(lhs_expr, rhs_expr)` | Compile and register a function definition equation |
| `eval_expr(expr)` | Type-check and evaluate an expression; print result |
| `wr_expr(expr, file)` | Lazy-stream expression output |
| `display()` | Print current session definitions |
| `mod_use(name)` | Load a named module |
| `mod_save(name)` | Save session to a module file |
| `mod_private()` | Mark subsequent definitions as private |
| `clean_slate()` | Reset per-statement state after each `;` |
| `mod_fetch()` | Commit current statement's changes to the module |
| `preserve()` | Mark current heap state as permanent (for Standard module) |

### Error recovery

When a syntax error is encountered, YACC discards tokens until it finds `;`, then `yyerrok` resets the error state. During recovery in interactive mode, `yylex()` synthesises a `;` at end-of-line to help resynchronisation.

## Operator System (`src/op.h`)

Operator precedences range from `MINPREC=1` (weakest) to `MAXPREC=9` (strongest). Associativity is `ASSOC_LEFT` or `ASSOC_RIGHT`.

Operators are stored in a hash table (`table.c`). `op_declare(name, prec, assoc)` inserts; `op_lookup(name)` retrieves.

The Standard module pre-declares these operators (from `lib/Standard.hop`):

| Operators | Precedence | Assoc |
|-----------|-----------|-------|
| `->` | 2 | right |
| `o` | 2 | left |
| `#`, `X` | 4 | right |
| `or` | 1 | left |
| `and` | 2 | left |
| `::` | 5 | right |
| `<>` | 5 | right |
| `=`, `/=` | 3 | left |
| `<`, `=<`, `>`, `>=` | 4 | left |
| `+`, `-` | 5 | left |
| `*`, `/`, `div`, `mod` | 6 | left |

## Implications for the C++20 Rewrite

The YACC-based parser is the most deeply platform-entangled part of the codebase. Replacing it with a hand-written recursive-descent parser in C++20 will:

1. **Eliminate** the yacc/bison, AWK, and sed build-time dependencies.
2. Enable **source location tracking** (file/line/column) at every AST node — currently impossible with YACC actions.
3. Allow **better error messages**: the parser can describe what it expected vs. what it found.
4. Allow **incremental parsing** for the REPL without re-parsing the whole input.

The operator precedence climb (`BINARY` token encoding) will become a standard **Pratt parser** or **precedence climbing** approach, where the operator table is consulted at parse time to decide whether to keep parsing or return.

See [11-rewrite-roadmap.md](11-rewrite-roadmap.md) for the full plan.
